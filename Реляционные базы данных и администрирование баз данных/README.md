# Домашнее задание к занятию "`Резервное копирование баз данных`" - `Гаврилова Валерия`

### Задание 1

1.1. Восстановление данных в полном объёме за предыдущий день

Для этого сценария оптимален классический подход: Полная резервная копия (Full Backup), создаваемая раз в сутки.
В запланированное время (например, в 02:00 ночи) создаётся полный снимок всей базы данных.
При сбое система разворачивает последнюю полную копию от предыдущей ночи. Потери данных составят от нескольких часов до суток (всё, что изменилось после создания копии).
Чтобы минимизировать потери, можно комбинировать с дифференциальными копиями (копируются только изменения с момента последнего полного бэкапа). Тогда восстановление будет таким: последняя полная копия + последняя дифференциальная.
Однако данные за текущий день (с момента последнего бэкапа до сбоя) будут потеряны.

1.2. Восстановление данных за час до предполагаемой поломки

Здесь требуется гораздо более частая фиксация изменений. Подойдут два варианта:

Вариант первый (классика): Полная копия + Журналы транзакций (Transaction Log Backup) каждые 15–30 минут.
Раз в сутки делается полная копия. А в течение дня — каждые 15–60 минут — бэкап только журнала транзакций (всех изменений, коммитов, операций).
Разворачивается последняя полная копия, затем последовательно накатываются все журналы транзакций вплоть до нужного момента времени (point-in-time recovery), например «за час до сбоя» или «за 5 минут до удаления важной таблицы».

Точность: До секунды (смотря с какой периодичностью делаются бэкапы логов).

Вариант второй (современный): Инкрементные копии с высокой частотой (например, каждые 30 минут).
Инкрементный бэкап копирует только изменения с момента последнего любого бэкапа (полного или инкрементного). Это быстрее, чем бэкап логов, но восстановление сложнее (нужно накатывать цепочку инкрементов).

Для восстановления «за час до» — критически важны частые бэкапы логов транзакций или частые инкрементные копии.

1.3. Кейс с моментальным переключением на работающую / починенную базу

Да, такой кейс возможен, и он называется высокая доступность (High Availability, HA) и отказоустойчивость. 
Варианты реализации:

- Синхронная репликация (кластер с автоматическим переключением, failover): Есть как минимум два узла: основной (primary) и реплика (standby). Каждая транзакция записывается на оба узла синхронно (применяется только после подтверждения от реплики). При поломке основного узла система автоматически (или по команде администратора) переключается на реплику моментально. Потери данных = 0.

Примеры: Always On Availability Groups (Microsoft SQL Server), Patroni/PostgreSQL с синхронной репликацией, Galera Cluster (MySQL/MariaDB).

- Асинхронная репликация + быстрый failover: Реплика отстаёт на несколько секунд. При сбое основного узла переключение происходит моментально, но могут потеряться последние несколько транзакций (несколько секунд данных).

- Хранилище с «мгновенными снимками» (Snapshot/Split-mirror): Используется дисковый массив (SAN), который создаёт мгновенный «слепок» базы. При сбое система переключается на заранее созданный слепок. Это не секунды, но минуты. Быстрее, чем восстанавливать из лент, но не «моментально».

Ключевое отличие от бэкапа: Бэкап — это копия данных на случай катастрофы (восстановление занимает часы). А здесь — горячая резервная копия (replica) с автоматическим переключением, которая позволяет продолжить работу почти без простоя (секунды).

Выводы: Кейс с моментальным переключением на работающую базу возможен. Это кластеризация + репликация, а не традиционное резервное копирование. Часто используется для критичных финансовых систем, где недоступность даже на 5 минут недопустима.
